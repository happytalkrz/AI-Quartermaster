# AQM v0.2 통합 개선 플랜

> 작성일: 2026-03-28
> 기준: v0.1.0 (KGB 평가 6.5/10)
> 목표: KGB 8.0+ / 실사용 가능한 안정성 확보

---

## 핵심 문제 인식

**"실패 = 사망" 문제**: 현재 파이프라인은 어떤 단계에서든 실패하면 즉시 FAILED로 전환되고 끝난다.
복구 메커니즘이 전혀 없어서 인간이 직접 개입해야 한다. 이것이 AQM의 가장 큰 구조적 약점이다.

이 문제는 단일 이슈로 해결되지 않고 여러 개선의 조합으로 점진적으로 완화된다:
- Phase 재시도 (#1) → Phase 레벨 자동 복구
- 에러 리포팅 (#3) → 실패 시 인간 개입 비용 최소화
- 환경 사전 검증 (#4) → 환경 문제로 인한 실패 사전 차단
- **[장기] 파이프라인 재개 기능** → 실패 지점부터 이어서 실행 (아래 기록)

---

## 장기 과제 기록 (현재 플랜 범위 밖)

### LT-1: 파이프라인 재개 (Resume from failure)

**문제**: Phase 3/5에서 실패하면 Phase 1~2의 작업물(커밋, worktree)은 남아있지만
파이프라인을 처음부터 다시 돌려야 한다. 대형 이슈에서 시간과 API 비용 낭비.

**필요한 것**:
- 파이프라인 상태를 JSON으로 체크포인트 저장 (각 state 전환 시)
- `aqm resume --job <id>` 명령으로 마지막 성공 state부터 재개
- worktree와 branch가 살아있는지 검증 후 이어서 실행
- Phase 결과 배열을 체크포인트에서 복원

**난이도**: 높음 (orchestrator 구조 변경, 상태 직렬화/역직렬화 필요)
**선행 조건**: Phase 재시도(#1), 에러 리포팅(#3) 완료 후 착수 권장

### LT-2: 파이프라인 부분 롤백

**문제**: Phase 3에서 실패했을 때, Phase 1~2 커밋을 유지할지 롤백할지 선택 불가.
현재 rollback-manager.ts가 있지만 orchestrator에서 호출하지 않음.

**필요한 것**:
- 실패 시 rollback 전략 설정: `none` | `all` | `failed-only`
- rollback-manager를 orchestrator의 FAILED 핸들러에 연결
- 이슈 코멘트에 롤백 여부 포함

## 작업 순서 (1단계 내)

```
1-3 (stuck timeout) ← 가장 간단, 워밍업
  ↓
1-2 (maxPhases 반영) ← 5줄 수정
  ↓
1-4 (에러 리포팅) ← 1-1의 기반
  ↓
1-1 (Phase 재시도) ← 핵심, 1-4 에러 분류 활용
  ↓
1-5 (CLAUDE.md 주입) ← 독립적, 마지막
```

1-3 → 1-2는 병렬 가능.
1-4 → 1-1은 순차 (에러 분류가 재시도 로직에 필요).
