# AQM 이슈 생성

## 이슈 작성 규칙

AQM 파이프라인이 처리할 이슈는 아래 형식을 따른다.

### 필수 구조

```markdown
## 요구사항

- [ ] 구체적 요구사항 1
- [ ] 구체적 요구사항 2

## 관련 파일

- `src/path/to/file.ts` — 설명
- `src/path/to/file2.ts` — 설명

## 참고

- 기존 동작을 깨뜨리지 않을 것
- `npx tsc --noEmit` + `npx vitest run` 통과 필수
```

### 이슈 크기 기준

AQM은 Phase별 타임아웃과 리뷰 검증(analyst)이 있으므로 이슈 크기가 성공률에 직결된다.

**최적 크기** (성공률 최고):
- 요구사항 2~3개
- 수정 파일 2개
- Plan 결과 Phase 2~3개
- 하나의 명확한 목적

**적정 크기** (성공 가능):
- 요구사항 3~4개
- 수정 파일 3개
- Plan 결과 Phase 3~4개

**너무 큰 이슈** (실패 위험 높음 — 반드시 분할):
- 요구사항 5개 이상
- 수정 파일 4개 이상
- "A도 하고 B도 하고 C도 하라" 식의 복합 목적
- 여러 모듈에 걸친 아키텍처 변경

**분할 원칙**:
- 기능 단위로 쪼갠다 (타입 정의 → 핵심 로직 → 연동이 아니라, 기능A → 기능B)
- 각 이슈가 독립적으로 머지 가능해야 한다
- 의존성 있으면 `depends: #N`으로 순서 명시
- 타입/유틸리티를 먼저 분리하면 후속 이슈가 단순해진다

**실패 사례 (크기 관련)**:
- #54 (자동 복구 파이프라인, 요구사항 8개) — 4회 연속 실패 (타임아웃 + 리뷰 실패)
- #63/#64 (2분할, 요구사항 4~5개) — 리뷰 분석 실패 (analyst가 미충족 판정)
- #65~#68 (4분할, 요구사항 2~3개) — 최적 크기로 재분할하여 성공 목표
- **교훈**: 요구사항 5개 이상이면 반드시 분할. 3개 이하가 안전.

### UI/디자인 이슈 작성 규칙

UI 변경이 포함된 이슈는 반드시 아래 규칙을 따른다:

- **Stitch 디자인 파일이 `docs/design/`에 있으면** 이슈 본문에 참조 경로를 명시
- **디자인 파일이 없으면** 이슈 생성 전에 사용자에게 Stitch에서 디자인을 먼저 받아올 것을 요청
- **임의 디자인 금지** — AQM이 자체적으로 UI를 디자인하지 않는다
- 이슈 본문 예시:
  ```markdown
  ## 참고
  - UI 디자인: `docs/design/settings-page-stitch.html` 참조
  - Stitch 디자인 시스템 기반으로 구현
  ```

### 좋은 이슈 vs 나쁜 이슈

**좋은 이슈** (AQM이 잘 처리함):
- 명확한 입출력이 있는 기능 추가
- 특정 파일의 특정 함수 수정
- 설정 필드 추가 (3곳 동시 수정: types, defaults, validator)
- 테스트 추가
- 버그 수정 (에러 메시지와 재현 경로 포함)

**나쁜 이슈** (사람이 해야 함):
- "전체 리팩터링"
- "성능 개선" (구체적 지표 없이)
- 여러 모듈에 걸친 아키텍처 변경
- UI/UX 주관적 판단이 필요한 것

### 의존성 표기

다른 이슈가 먼저 완료되어야 하면:
```
depends: #42
```

### 라벨

이슈 생성 후 `ai-quartermaster` 라벨을 붙여야 AQM이 처리한다.

## Gotchas
- [2026-04-04] 요구사항 5개 이상 이슈는 거의 항상 실패 — analyst 리뷰 미충족 판정 또는 타임아웃. 최적은 2-3개, 4개까지 허용 (#54, #63, #64)
- [2026-04-04] UI 이슈에 Stitch 디자인 파일 없으면 AQM이 임의 디자인해서 품질 저하 — 이슈 생성 전 반드시 `docs/design/`에 디자인 파일 확인

## 실행 방법

사용자에게 이슈 내용을 확인시킨 후:

```bash
gh issue create --repo happytalkrz/AI-Quartermaster \
  --title "이슈 제목" \
  --body "이슈 본문" \
  --label "ai-quartermaster"
```

**주의**: `--label` 붙이면 바로 AQM이 처리 시작한다. 검토 후 라벨을 붙이고 싶으면 `--label` 빼고 생성 후 수동으로 라벨 추가.

## 이슈 제목 컨벤션

```
feat: 기능 추가 설명
fix: 버그 수정 설명
refactor: 리팩터링 설명
test: 테스트 추가 설명
docs: 문서 수정 설명
```
